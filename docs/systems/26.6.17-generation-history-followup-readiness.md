# 26.6.17 generation-history follow-up readiness

이 문서는 `26.6.17 feature-refactor-followup` 로드맵에서 생성 이력 화면과 재실행 경로를 로컬 검토 가능한 상태로 묶는 운영 메모입니다. package/app version bump, push, deploy, restart, protected service `3999` 조작은 포함하지 않습니다.

## 변경 범위

- 실패하거나 취소된 생성 이력을 개별, 선택, bulk 단위로 다시 실행할 수 있게 재실행 진입점을 맞췄습니다.
- 생성 이력 결과 카드에서 해상도, 파일 형식, 모델 정보를 작은 media review badge로 확인할 수 있게 했습니다.
- generation history panel이 queue retry API를 직접 소유하지 않도록 `generation-history-retry-actions.ts`로 재실행 경계를 분리했습니다.
- retry 대상 queue job id는 helper에서 중복 제거한 뒤 queue retry mutation으로 넘기도록 정렬했습니다.

## 운영 확인 포인트

1. 실패/취소 이력이 없으면 bulk rerun 버튼은 실행 대상으로 보이지 않아야 합니다.
2. 선택한 이력만 다시 실행할 때 retry 가능한 queue job id만 전달되어야 합니다.
3. media badge는 파일을 다시 처리하거나 metadata를 변경하지 않고, 이미 받은 history/result 데이터를 표시하는 용도입니다.
4. cleanup, deletion, retention policy, auth/security, public API, live demo 작업은 이 로드맵 밖 승인 경계로 남습니다.

## 로컬 검증

이번 follow-up의 기준 확인은 아래 명령으로 남깁니다.

```bash
npm run verify:generation-history-feed-progress-ui-contracts
npm run verify:generation-queue-ui-contracts
npm run verify:release-readiness
python -m graphify update .
git diff --check
```

`verify:release-readiness`는 root alias로 유지하며, 내부에서 workspace script alias drift, VitePress 문서 빌드, 전체 build를 순서대로 확인합니다. Graphify update는 이전 코드 변경 이후 knowledge graph를 현재 commit 기준으로 맞추기 위한 로컬 갱신입니다.

## 승인 경계

- Git push, release tag, public announcement
- live demo host pull, deploy, service restart
- protected service `3999` 조작
- DB schema, retention policy, destructive cleanup
- auth/security policy change or public API expansion
- package/app version bump
