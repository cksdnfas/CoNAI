이 문서는 `26.6.8 hardening-evidence-observability` 로드맵의 최종 로컬 준비 기록입니다. 범위는 dependency/security hardening, operator evidence automation, media/runtime observability를 하나의 검토 흐름으로 묶는 데까지입니다. push, deploy, live demo update, server restart, protected service `3999`, package version bump, git tag, public release action은 실행하지 않았습니다.

## 완료된 축

- Dependency/security hardening: safe lane dependency update와 `npm audit` triage를 완료했고, 승인 없이 처리하면 위험한 후보를 분리했습니다.
- Operator evidence automation: release readiness workspace가 로컬 히스토리 저장, 이전 기록 복원, Markdown handoff copy/download를 제공합니다.
- Media/runtime observability: media review와 workflow runner가 queue, quality backlog, similarity decision, cleanup staging, retry, recovery, retention, terminal run trend를 보여줍니다.
- Threshold guidance: 기존 관측 데이터 안에서만 operator-visible threshold와 approval boundary를 표시합니다.

## 임계값 안내

Media review threshold guidance:

- `review-queue-threshold`: session 미검토 항목 `48`개부터 attention 기준으로 둡니다.
- `quality-backlog-threshold`: missing/sparse tags, unrated, ungrouped 합산 `24`개부터 attention 기준으로 둡니다.
- `similarity-review-threshold`: needs-human-review similarity decision `1`개부터 다음 cleanup 판단 전 검토 대상으로 둡니다.
- `cleanup-approval-threshold`: cleanup staging `1`개 이상이면 review record까지만 만들고 destructive cleanup이나 retention policy 변경은 별도 승인 대상으로 둡니다.

Workflow runtime threshold guidance:

- `queue-pressure-threshold`: active queue가 schedule concurrency limit을 넘거나 cancellation/process mismatch가 있으면 operator review 대상으로 둡니다.
- `retry-stop-threshold`: paused/stopped autorun `1`개부터 stop reason review 대상으로 둡니다.
- `recovery-mismatch-threshold`: startup recovery/process mismatch `1`개부터 recent failure와 output state를 같이 봅니다.
- `retention-approval-threshold`: pending retention prune `1`개부터 approval-required boundary로 둡니다.
- `terminal-failure-threshold`: failed terminal run 비율 `20%`부터 rerun 전 error/input diff review 대상으로 둡니다.
