# Module Workflow Output Management Plan

## Goal

Add a folder-aware management surface for outputs generated by module workflows.

The new system should let users:
- browse generated images, videos, and other output artifacts with the existing list/grid UI patterns
- select multiple generated outputs and download them
- copy selected outputs into managed content folders used by the main library workflow
- review and clean up executions or queued jobs that produced no usable outputs

## UX direction

### Recommended placement

Use the browse-mode content area of the module workflow screen.

Why:
- the left workflow/folder sidebar already establishes a folder-centered mental model
- the right browse content area currently has no strong folder-specific content when only a folder is selected
- output management is naturally scoped by folder and its descendant workflows

### Scope behavior

- **Root selected**
  - show an overall inbox-style summary for recent generated outputs
  - show orphan/empty executions that still need cleanup
- **Folder selected**
  - show outputs generated by workflows inside the selected folder subtree
  - show execution queues and empty runs for the same subtree

## Product shape

### Main tabs

Inside browse mode, add a content panel with two tabs:

1. **Generated Outputs**
   - list/grid of image/video/file artifacts
   - reuse the existing image/media selection patterns where possible
   - support multi-select, download, and copy-to-folder

2. **Queue / Empty Runs**
   - list queued, running, failed, cancelled, and completed-with-no-output executions
   - support cleanup actions for useless runs

## Data model assumptions

Current backend already has:
- workflow folders (`graph_workflow_folders`)
- workflows (`graph_workflows`) with `folder_id`
- executions (`graph_executions`)
- execution artifacts (`graph_execution_artifacts`)
- final results (`graph_execution_final_results`)

Current gap:
- no folder-scoped browse endpoint that aggregates workflows, executions, artifacts, and empty runs
- no copy/download management endpoint for generated artifacts

## Proposed backend additions

### Phase 1: browse foundation

Add a folder-scoped browse endpoint that returns:
- selected scope metadata
- workflows in the selected folder subtree
- executions for those workflows
- artifacts for those executions
- final results for those executions
- empty executions summary

Suggested endpoint:
- `GET /api/graph-workflows/browse-content?folder_id=<id>`
- allow missing `folder_id` for root scope

### Phase 2: management actions

Add endpoints for:
- bulk artifact download (zip or streamed batch)
- copy selected artifacts into a managed folder target
- bulk cleanup of empty or stale executions

## Proposed frontend additions

### Phase 1

- add browse-content query hook for folder/root scope
- render folder-aware browse content in module workflow browse mode
- reuse existing media list and selection UI patterns
- show separate empty-run management list

### Phase 2

- add selection action bar for generated outputs
- add bulk download
- add copy-to-folder target picker
- add cleanup actions for empty runs

## Implementation notes

- keep phase 1 read-heavy and safe
- do not mix artifact copy and library DB registration in the first slice
- prefer file copy + existing scanner integration before attempting direct metadata insertion
- treat final results as the preferred user-facing output signal, but keep raw artifacts available for debugging
- expose enough execution metadata for cleanup decisions without opening full execution detail every time

## Success criteria

### Phase 1 success

- backend exposes one folder-aware browse-content endpoint
- root scope and folder scope both resolve without errors
- response includes workflows, executions, artifacts, final results, and empty execution summaries
- backend build passes

### Phase 2 success

- browse UI renders folder-scoped generated outputs in the existing module workflow browse view
- users can select multiple outputs
- users can download selected outputs
- users can identify and clean up empty runs
